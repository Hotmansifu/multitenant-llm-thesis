# Split-Process Architecture (RQ3)

Corresponds to **§3.2, §3.3, §4.3** of the thesis.

## Research Question

- **RQ3:** How can a split-process architecture protect users on an untrusted server,
  and what is the speed penalty?

## Environment

- Bare metal hardware
- Single CPU thread: `OMP_NUM_THREADS=1` (required for consistent results)
- All demos must be run with `OMP_NUM_THREADS=1`

## Architecture

The transformer computation is split between a trusted client and an untrusted server.
Split point: between attention layers (client) and MLP layers (server).
Hidden state shape per transfer: T×C where T=8, C=384 → 12 KB per transfer, 144 KB per forward pass.

Threat model: honest-but-curious server owner (§3.2).

## Files

| File | Description |
|---|---|
| `demo1_shm.py` | D1 — shared memory split (plaintext baseline) |
| `demo2_net.py` | D2 — TCP localhost split (plaintext baseline) |
| `demo3_phe.py` | D3 — TCP + Paillier homomorphic encryption |
| `model.py` | GPT model with Gaussian kernel attention (σ=4) |

## Results

| Implementation | Overhead | Notes |
|---|---|---|
| D1 — shared memory | ~40% | Two runs: 35.6% and 41.4% |
| D2 — TCP localhost | ~560% | Two runs: 551% and 577% |
| D3 — TCP + Paillier | ~56,000× | Extrapolated from 32-element benchmark |

Texts generated by split and monolithic baseline are identical in all cases ✅

## Running

```bash
OMP_NUM_THREADS=1 python3 demo1_shm.py
OMP_NUM_THREADS=1 python3 demo2_net.py
OMP_NUM_THREADS=1 python3 demo3_phe.py  # takes several minutes
```

## Notes

- Timing varies with machine load — original thesis measurements shown above
- D3 overhead is extrapolated from a 32-element Paillier benchmark, not a full forward pass (§3.3)
- Server process uses `multiprocessing.Process` with spawn start method (avoids PyTorch fork issue on Linux)
- Communication in D1 uses one `/dev/shm` memory file and a `multiprocessing.Pipe` for synchronisation
